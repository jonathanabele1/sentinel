# uv

## What it is

`uv` is a Python package manager and project runner written in Rust. It replaces the entire traditional Python stack of `pip` + `pip-tools` + `virtualenv` + `pyenv` with a single binary. It's roughly 10-100x faster than the old stack because Rust + parallel downloads + a smarter resolver, and it gives you a real lockfile out of the box.

For most projects starting in 2025, `uv` is the obvious choice.

## What it does

Five jobs, all in one tool:

1. **Installs Python itself.** `uv` will download and manage Python versions for you. No need for `pyenv` or asdf or system Python.
2. **Creates and manages virtualenvs.** Per-project `.venv/` folder, automatic when you run any `uv` command.
3. **Resolves and installs dependencies** from `pyproject.toml`, writes a lockfile (`uv.lock`).
4. **Runs commands inside the project's venv** via `uv run <cmd>`.
5. **Manages dev vs prod dependency groups** via `[dependency-groups]` in `pyproject.toml`.

## What replaces what

| Old way | uv way |
| --- | --- |
| `pyenv install 3.12 && pyenv local 3.12` | `.python-version` file; uv installs 3.12 automatically |
| `python -m venv .venv` | uv creates `.venv` automatically on first `uv sync` |
| `source .venv/bin/activate` | `uv run <cmd>` (no activation needed) |
| `pip install -r requirements.txt` | `uv sync` (installs from the lockfile) |
| `pip install requests` | `uv add requests` (also updates `pyproject.toml` and `uv.lock`) |
| `pip install -e .` | `uv sync` (editable install of your project is automatic) |
| `pip freeze > requirements.txt` | `uv.lock` (autogenerated, deterministic) |
| `pip-compile` (pip-tools) | Built in to `uv lock` |

The mental shift: with the old stack, *you* manage the venv and pip just installs into it. With `uv`, the venv is an implementation detail; you express what you want in `pyproject.toml` and `uv` makes the venv reflect that.

## Files in a uv project

Three files in the repo root drive uv:

```
pyproject.toml        Declares dependencies, dev dependencies, build config, tool configs.
uv.lock               Locked, deterministic resolution. Commit this; humans don't edit it.
.python-version       Single line: "3.12". Tells uv which Python interpreter to use.
```

The `.venv/` folder is gitignored. It's regenerated by `uv sync` from the lockfile.

## How Sentinel uses it

Look at `pyproject.toml`:

```toml
[project]
name = "sentinel"
requires-python = ">=3.12"
dependencies = [
    "fastapi>=0.115.0",
    "uvicorn[standard]>=0.32.0",
    "pydantic>=2.9.0",
    # ...
]

[dependency-groups]
dev = [
    "pytest>=8.3.0",
    "ruff>=0.7.0",
    "mypy>=1.13.0",
    # ...
]
```

- **`dependencies`** are what gets installed in production: FastAPI, SQLAlchemy, the Anthropic SDK, everything the running service needs.
- **`[dependency-groups].dev`** are the tools used during development: pytest, ruff, mypy, pre-commit. They are not installed in production builds.

The `>=` version specifiers say "at least this version, anything newer is fine." The lockfile pins the exact resolved version so two developers running `uv sync` get bit-for-bit identical environments.

## Common commands

```bash
# Install everything from the lockfile into .venv. Idempotent; safe to run any time.
uv sync

# Install everything including dev deps (default behaviour anyway).
uv sync --all-groups

# Add a new dependency. Updates pyproject.toml AND uv.lock atomically.
uv add httpx

# Add a dev-only dependency.
uv add --dev pytest-mock

# Remove a dependency.
uv remove httpx

# Refresh the lockfile without installing (e.g. after manually editing pyproject.toml).
uv lock

# Run a command inside the project's venv. The killer feature.
uv run pytest
uv run mypy
uv run python -c "import fastapi; print(fastapi.__version__)"

# Frozen mode: refuse to update the lockfile (used in CI).
uv sync --frozen
```

## `uv run` is the part that matters day to day

`uv run X` does roughly this:

1. Find the project's `pyproject.toml` (walks up from the current directory).
2. Make sure `.venv/` exists and is up to date with the lockfile (re-syncs if not).
3. Run `X` with the venv's Python and the venv's installed packages on the path.
4. Exit with whatever exit code `X` exited with.

So `uv run pytest` is equivalent to `source .venv/bin/activate && pytest && deactivate`, except faster and without you having to remember the activate/deactivate dance.

This is why our Makefile prefixes everything with `uv run`. The Makefile target `lint` is literally `uv run ruff check .`. It works the same whether you typed `make lint` from a freshly cloned repo with no venv set up (uv creates one) or from a long-running shell (uv reuses the existing venv).

## Lockfiles, briefly

A lockfile pins **every** package to **exact** versions, including transitive dependencies. Without one:

```
You today:        pip install fastapi (gets 0.115.0)
You in 6 months:  pip install fastapi (gets 0.118.0, breaks something)
A teammate:       pip install fastapi (gets 0.116.0, slightly different)
```

With `uv.lock`:

```
Everyone:         uv sync (gets exactly fastapi==0.115.0 plus the same starlette, pydantic-core, etc.)
```

The lockfile is committed to the repo. CI runs `uv sync --frozen` so it refuses to silently upgrade. You only get new versions when you explicitly run `uv add` or `uv lock`.

## Why we use it

1. **Speed.** A full `uv sync` from scratch is a couple of seconds. `pip install -r requirements.txt` for the same dep list is 30 seconds plus.
2. **One tool instead of four.** No more cycling through pyenv → venv → pip → pip-tools.
3. **Real lockfiles built in.** With pip you had to add `pip-tools` to get this. With uv it's automatic.
4. **Project-rooted commands.** `uv run` works from anywhere in the project tree without remembering to activate the venv.

## Common confusions

**"Do I need to activate the venv?"** No. `uv run X` handles it. If you *want* an activated shell for poking around, run `source .venv/bin/activate`. But the daily workflow doesn't need it.

**"Why is there a .venv folder?"** It's the standard Python virtualenv layout. `uv` creates it like any venv tool would; the difference is `uv` manages its contents from the lockfile.

**"Do I commit `uv.lock`?"** Yes. Always. It's the entire point of having a lockfile.

**"Do I commit `.venv/`?"** No. It's gitignored; it's machine-specific and reproducible from the lockfile.

**"What if my teammate doesn't have uv installed?"** Install it: `brew install uv` or `curl -LsSf https://astral.sh/uv/install.sh | sh`. Five seconds.

## TL;DR

`uv` is one Rust binary that handles Python installation, virtualenvs, dependency resolution, lockfiles, and running commands inside the venv. It replaces pip + pip-tools + virtualenv + pyenv. The daily commands you'll use are `uv sync` (set up the env) and `uv run X` (run X inside the env). The lockfile (`uv.lock`) is committed; the venv (`.venv/`) is not.

2. Why is `uv.lock` committed to git but `.venv/` is not?
3. The CI workflow runs `uv sync --frozen`. What does the `--frozen` flag change, and why does CI want it?
4. You're adding a new dependency. What's wrong with `pip install requests` in a uv-managed project, even if you remember to update `pyproject.toml` by hand?
5. What goes in `[project].dependencies` vs `[dependency-groups].dev` in `pyproject.toml`? Why does the distinction matter?
6. How would you investigate "this works on my laptop but fails in CI" when the codebase is identical? Where does uv help you?
